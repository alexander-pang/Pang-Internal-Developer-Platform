
# ADR-001: Use k3d for local Kubernetes

## Status
Accepted

## Context
We need a local Kubernetes cluster for development. Options: kind, k3d, minikube, Docker Desktop K8s.

## Decision
Use k3d.

## Reasoning                                                                                            
- Faster startup — important when iterating
- Lower memory footprint — running 6+ components on a laptop
- Built-in load balancer simplifies local ingress
- k3s differences from vanilla K8s are not relevant to our use case
- running 6 components on mac simultaneously
## Tradeoffs
- k3s is not vanilla Kubernetes — some edge cases may differ in production
- Traefik included by default adds noise (disable it)
- when k3s difference mattered

## Consequences
All cluster setup docs reference k3d. If production uses EKS/GKE, subtle
differences should be documented when moving from local to production.
if we hit an incompatability, we switch to kind
